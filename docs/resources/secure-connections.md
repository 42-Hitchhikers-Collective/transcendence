# Secure Connections (HTTPS)

## Requirement

> All communication between frontend and backend must be encrypted using HTTPS.

## Why TLS termination at nginx?

We chose **TLS termination** — nginx handles all HTTPS encryption/decryption, and forwards plain HTTP to the backend over Docker's internal network. Three reasons:

1. **Single responsibility** — the API server (Fastify) focuses on business logic, not crypto operations. nginx is purpose-built for TLS and does it efficiently.
2. **Simplified backend** — no need to configure SSL in Fastify or manage certs in the API code. The API just serves HTTP on port 3000.
3. **Industry standard** — this is how virtually every production deployment works (AWS ALB, Cloudflare, Nginx, etc.). TLS at the edge, plain HTTP internally.

The internal Docker bridge traffic (`nginx → api:3000`) never leaves the host machine. It's not routable, not exposed, and not observable by anyone outside the host. The subject only cares about what goes over the wire between client and server — which is 100% HTTPS.

## Port mapping

| Host port | Container port | Protocol | Purpose |
|-----------|---------------|----------|---------|
| `8443` | `443` | HTTPS (TLS) | Main application entry point |
| `80` | `80` | HTTP | Redirect → `https://localhost:8443` (UX convenience only) |

### Why the HTTP redirect on port 80?

It's a **UX convenience**, not a subject requirement. When a user types `localhost` in the browser, the browser defaults to `http://localhost:80`. Without the redirect they'd see "connection refused" and think the app is broken. With it, they get seamlessly forwarded to `https://localhost:8443`.

The 301 redirect contains zero application data — just `Location: https://...`. All actual API calls, socket messages, auth tokens, and game data flow exclusively over HTTPS. The redirect can be removed entirely with no impact on the app's functionality or compliance.

## nginx configuration

```nginx
# HTTP → HTTPS redirect (UX only, safe to remove)
server {
    listen 80;
    server_name localhost;
    return 301 https://$host:8443$request_uri;
}

# HTTPS (TLS termination — this is the real entry point)
server {
    listen 443 ssl;
    server_name localhost;

    ssl_certificate     /etc/nginx/certs/dev.crt;
    ssl_certificate_key /etc/nginx/certs/dev.key;

    location /api/     { proxy_pass http://api_upstream; ... }
    location /socket.io/ { proxy_pass http://api_upstream; ... }
    location /         { proxy_pass http://web_upstream; ... }
}
```

## Frontend

All frontend requests use relative paths (`/api/*`, `/socket.io/*`) with `credentials: "include"` — every request is automatically sent over the same HTTPS connection the page was loaded on. No `http://` URLs in the codebase, no mixed content risk.

## SSL certificates

Self-signed certificates for local development, generated by `make certs`:

```
nginx/certs/dev.crt   # Certificate
nginx/certs/dev.key   # Private key
```

For eval on other devices, use `make host-certs` to include the LAN IP in the cert's Subject Alternative Names (SAN).

## How to test / verify

### 1. Browser address bar
Open `https://localhost:8443` — the address bar shows `https://` and a lock icon (may show "not secure" with self-signed cert, accept it).

### 2. DevTools Network tab
Open DevTools → Network → refresh the page. Every request in the list should use the `https://` protocol. Check:
- `/api/*` calls → `https://localhost:8443/api/...`
- `/socket.io/*` → `wss://` (WebSocket Secure) or `https://`
- No `http://` entries anywhere

### 3. Mixed content check
Open DevTools → Console. There should be **zero** mixed content warnings. If you see "Mixed Content: The page was loaded over HTTPS but requested an insecure..." something is wrong.

### 4. Confirm HTTP port works (optional)
Visit `http://localhost` — you should be instantly redirected to `https://localhost:8443`. This confirms the UX redirect is functional but does not affect the encryption requirement.

### 5. Confirm HTTP on HTTPS port fails (expected)
Visit `http://localhost:8443` — this should fail with an SSL error. This is **correct** behavior: port 8443 only speaks TLS. A plain HTTP request to a TLS port cannot be understood, so no redirect is possible here.
